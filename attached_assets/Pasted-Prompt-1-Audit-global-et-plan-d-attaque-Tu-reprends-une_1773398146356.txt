- sports :

  - rugby
  - handball
  - volleyball
  - football

- PostgreSQL + RLS + Edge Functions

   - org_display_settings
   - org_sport_settings

   - scheduled
   - live
   - paused
   - finished

   - 1 équipe = 1 URL stable
   - paramètre principal = teamSlug
5. ne jamais réintroduire :
   - token match
   - display_token
   - public_display
   - orgs.org_sport
   - orgs.display_settings

- proposer un plan d’attaque incrémental sans refonte

Je veux :
1. un audit court des reliquats legacy
2. la liste des fichiers critiques à corriger
3. la liste des tables / edge functions critiques
4. un plan en 5 étapes maximum
5. aucune refonte gratuite

Important :
- pars de l’existant
- ne propose pas d’architecture abstraite
- reste pragmatique
- dis précisément quels fichiers doivent être modifiés
Prompt 2 — Stabilisation complète Display + Edge Function
Stabilise complètement le flux Operator -> Display sur scoreDisplay V2.

Contexte :
- Display public officiel = URL stable par équipe
- paramètre principal = teamSlug
- plus aucun mode public par token
- backend Supabase
- edge function critique = get-display-context
- source live = persist DB + broadcast + fallback postgres_changes

Objectifs fonctionnels :
- score quasi instantané côté Display
- chrono cohérent
- bouton Pause fonctionnel
- reset chrono fonctionnel
- changement de match correct en mode équipe stable
- affichage propre si aucun match n’est trouvé

Exigences techniques :
1. get-display-context :
   - supporte teamSlug, teamId, éventuellement matchId
   - utilise SUPABASE_SERVICE_ROLE_KEY
   - ne contient aucune occurrence de :
     - token
     - display_token
     - public_display
   - applique le fallback :
     - live
     - paused
     - scheduled
     - dernier finished/archived
2. Display main.tsx :
   - charge le contexte via get-display-context
   - écoute les broadcasts
   - écoute postgres_changes sur matches
   - recale l’horloge si clock_ms, clock_running ou match_id changent
   - refresh stable équipe plus fréquent que 15 secondes
3. Operator ControlPage :
   - start/pause/reset poussent les bonnes valeurs
   - priorité correcte des patchs live
   - DB + Display restent cohérents

Je veux :
- les fichiers complets modifiés
- le code complet de get-display-context si nécessaire
- les corrections exactes de main.tsx
- les corrections exactes de ControlPage.tsx
- un mini plan de test manuel

Contraintes :
- ne jamais casser basket / handball / volley / football
- rester compatible avec l’existant
- travailler par correction incrémentale
