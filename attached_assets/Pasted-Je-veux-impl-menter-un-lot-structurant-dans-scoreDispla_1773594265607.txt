Je veux implémenter un lot structurant dans scoreDisplay V2 : le CONTRAT LIVE GLOBAL.

Objectif produit :
toute information live diffusée entre Operator, QR Console et Display doit être protégée par live_seq.
Le chrono, en plus, doit être reconstruit à partir d’une ancre persistée.

Contexte existant :
- stack : React + TypeScript + Supabase
- 3 surfaces concernées :
  1. console principale Operator
  2. QR Console / MobileControlPage
  3. Display public
- des progrès ont déjà été faits sur le chrono :
  - live_seq existe à certains endroits
  - clock_anchor_epoch_ms / clock_anchor_clock_ms existent en DB
  - le Display et la QR Console savent déjà partiellement reconstruire le chrono à froid
- mais le contrat live n’est pas encore homogène sur toutes les informations :
  - score
  - statut
  - période
  - fautes
  - temps morts
  - bonus
  - shot clock
  - possession
  - cartons
  - sets
  - sanctions
  - remplacements
  - overlays

Règle à faire respecter désormais :
- toute mutation live doit :
  1. générer un live_seq monotone
  2. diffuser un patch live avec live_seq
  3. persister l’état en DB
  4. persister matches.last_event_seq = live_seq
  5. si la mutation concerne le chrono :
     - persister aussi clock_anchor_epoch_ms
     - persister aussi clock_anchor_clock_ms
- toute surface réceptrice doit :
  1. charger last_event_seq au démarrage
  2. ignorer tout patch avec live_seq < lastAppliedSeq
  3. appliquer uniquement les patches les plus récents
  4. reconstruire le chrono depuis l’ancre persistée si nécessaire

Je veux un vrai durcissement, pas une explication théorique.

1. génération du live_seq monotone
2. construction du patch live

5. mise à jour matches.last_event_seq
6. persistance éventuelle de l’ancre chrono
7. possibilité d’ajouter :
   - event log
   - overlay
8. valeur de retour claire pour l’appelant

Le but est que toutes les actions live passent par cette helper, pour arrêter d’avoir des logiques divergentes.
