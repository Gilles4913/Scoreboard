Prompt 2 — Correctif backend + operator sur la vérité métier des sin bins

Celui-ci est le plus important.

Tu vas maintenant implémenter le correctif minimal et robuste du flux rugby sin bin.

Objectif métier :
une exclusion temporaire rugby doit être correcte de bout en bout :
- création propre
- durée correcte basée sur le temps de jeu effectif
- pause gelée si clock_running = false
- disparition quand terminée
- cohérence entre la table match_sin_bins et les compteurs synthétiques du match
- pas de timer fantôme à 00:00

Contexte fonctionnel :
- show_sin_bin active l’affichage des exclusions rugby
- show_sin_bin_timer bascule entre affichage synthétique et timer détaillé
- les exclusions détaillées sont stockées dans public.match_sin_bins
- get-display-context calcule remaining_ms à partir des ancres chrono live
- le Display ne doit pas avoir à recalculer la vérité métier ; il consomme un payload propre
- le système live global avec last_event_seq existe déjà et ne doit pas être cassé

Bugs à corriger :
1. un sin bin arrivé à terme peut rester is_active=true et donc visible à 00:00
2. les boutons “-” jaune rugby peuvent désynchroniser les compteurs matches et la table match_sin_bins
3. la source de vérité doit devenir la table match_sin_bins active réelle, pas un simple compteur décrémenté à la main

Fichiers à modifier en priorité :
- apps/operator/src/pages/ControlPage.tsx
- supabase/functions/get-display-context/index.ts
- éventuellement apps/operator/src/realtime.ts si nécessaire pour typer le patch
- ajoute une migration SQL seulement si strictement nécessaire ; sinon évite

Travail demandé :

A. Corriger la lecture backend dans get-display-context
- au moment de charger match_sin_bins actifs, calcule remaining_ms comme aujourd’hui à partir de :
  - started_game_clock_ms
  - duration_s
  - temps de jeu effectif / ancre chrono
- filtre les sin bins expirés du payload détaillé :
  - si remaining_ms <= 0, ils ne doivent plus apparaître dans home_active_sin_bins / away_active_sin_bins
- recalcule les compteurs synthétiques rugby_home_sin_bin_active et rugby_away_sin_bin_active à partir des sin bins réellement encore actifs métier
- le payload retourné doit donc être cohérent même si les colonnes synthétiques du match ont dérivé
- ne fais pas d’effet de bord DB dans l’edge function si ce n’est pas nécessaire ; priorité à un payload exact

B. Corriger l’operator ControlPage
- identifie le code des boutons “-” sur les cartons jaunes rugby
- remplace le comportement actuel s’il décrémente directement rugby_home_yellow_sin_bin / rugby_home_sin_bin_active ou rugby_away_* sans toucher match_sin_bins
- nouveau comportement attendu :
  - pour rugby, le bouton “-” jaune doit retirer proprement la dernière exclusion active correspondante de l’équipe concernée, en clôturant une vraie ligne match_sin_bins
  - si aucune exclusion active n’existe, alors seulement décrémenter le compteur total de jaunes si le métier le permet
- garantis la cohérence entre :
  - match_sin_bins
  - rugby_home_yellow_sin_bin / rugby_away_yellow_sin_bin
  - rugby_home_sin_bin_active / rugby_away_sin_bin_active
- réutilise endRugbySinBin() si possible, ou factorise une fonction claire
- évite le code dupliqué home/away

C. Auto-nettoyage métier
- quand un sin bin est expiré selon le temps de jeu effectif, il ne doit plus être considéré actif
- au minimum, il doit disparaître du payload
- si tu peux le faire proprement sans fragiliser le système, ajoute aussi une clôture DB opportuniste côté operator quand on recharge l’écran ou quand on agit sur le match ; sinon laisse la DB telle quelle et rends le payload exact
- priorité absolue : exactitude fonctionnelle visible côté Display

Contraintes fortes :
- ne change pas l’API publique hors sujet
- ne touche pas au football
- ne touche pas aux overlays de remplacement
- ne touche pas au mode public teamSlug
- garde le style TypeScript existant
- garde des changements localisés
- si tu modifies une fonction, rends le fichier complet concerné

Livrables attendus :
1. résumé ultra-court des corrections
2. liste des fichiers modifiés
3. le contenu complet de chaque fichier modifié
4. explication précise de pourquoi le bug “timer bloqué à 00:00” est corrigé
5. explication précise de pourquoi le bug “bouton - désynchronise la table” est corrigé

Tests manuels à vérifier mentalement et à couvrir dans ta réponse :
- jaune rugby home sur joueur => timer visible
- pause chrono => timer figé
- reprise chrono => timer repart
- arrivée à terme => timer disparaît du payload détaillé
- bouton “Lever l’exclusion” => disparition correcte
- bouton “-” jaune rugby => plus de décrémentation aveugle qui laisse une ligne active orpheline
- deux exclusions simultanées même équipe => compteur et liste détaillée cohérents
Prompt 3 — Correctif Display pour mise à jour immédiate et disparition propre du timer

Celui-ci cible l’expérience temps réel côté public.

Tu vas maintenant corriger la partie Display du flux rugby sin bin pour que l’affichage soit propre en temps réel.

Objectif :
- quand une exclusion rugby est créée, le timer détaillé doit apparaître rapidement et proprement
- quand une exclusion est levée ou expirée, le timer doit disparaître proprement
- pas de timer fantôme affiché à 00:00
- pas de régression sur les autres sports

Contexte technique :
- le display charge un payload initial via get-display-context
- apps/display/src/main.tsx fusionne ensuite les données live
- apps/display/src/components/Scoreboard.tsx affiche soit :
  - le compteur synthétique d’exclusions
  - soit SinBinTimer quand show_sin_bin_timer = true
- le contrat live global basé sur last_event_seq existe déjà
- si le patch live ne contient pas les tableaux détaillés home_active_sin_bins / away_active_sin_bins, il faut gérer intelligemment la cohérence côté display sans casser le reste

Fichiers à inspecter et modifier si nécessaire :
- apps/display/src/main.tsx
- apps/display/src/components/Scoreboard.tsx
- éventuellement apps/display/src/api.ts

Bugs/risques à traiter :
1. le timer détaillé peut n’apparaître qu’après polling / refresh
2. après levée manuelle ou expiration, le timer peut rester visible trop longtemps
3. le composant SinBinTimer peut afficher 00:00 au lieu de disparaître
4. la fusion payload initial + patch live peut conserver une vieille liste détaillée

Ce que je veux :
A. sécuriser la fusion des données dans apps/display/src/main.tsx
- si un patch live modifie les compteurs synthétiques rugby_home_sin_bin_active / rugby_away_sin_bin_active mais ne contient pas les listes détaillées, éviter qu’une vieille liste détaillée reste affichée de façon incohérente
- la priorité métier :
  - si compteur actif = 0, la liste détaillée affichée pour ce côté doit être vidée
  - si compteur actif > 0 mais pas de détail récent, on peut laisser le polling recharger, mais jamais afficher de ligne impossible
- garde un comportement simple, robuste, lisible

B. sécuriser le rendu Scoreboard / SinBinTimer
- n’affiche jamais un timer dont remaining_ms <= 0
- si après filtrage il n’y a plus aucun sin bin valide, ne rends rien pour ce bloc timer
- le compteur synthétique doit rester possible si show_sin_bin_timer = false
- ne modifie pas le rendu des autres sports

C. garder la logique de pause
- clock_running = false => le timer visuel se fige
- clock_running = true => il décrémente
- mais un timer expiré ne doit plus rester visible à 00:00

Contraintes :
- pas de refactor massif
- pas de changement de style global
- pas d’impact sur football / basket / handball
- garde le composant lisible
- rends les fichiers complets modifiés

Livrables :
1. résumé court
2. fichiers modifiés
3. contenu complet de chaque fichier modifié
4. explication du scénario :
   - émission jaune rugby
   - affichage timer
   - expiration
   - disparition
   - levée manuelle
5. mention explicite que le bug “timer fantôme” est neutralisé
